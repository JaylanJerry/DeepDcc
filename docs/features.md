# 功能详解

## 定位

DeepDcc 面向的是“正式文档生产”而不是单次对话生成。它强调从输入任务到导出交付之间的整条工作流可控，而不是只比较模型回答质量。

## 核心能力

### 1. 任务解析与蓝图先行

系统会先理解任务意图，再给出结构化蓝图。用户可以在正文释放前确认、修改或重新生成章节结构，避免长文档一开始就失控。

### 2. 正文、预览、导出一致性导向

DeepDcc 的目标不是“编辑器有一份、预览再拼一份、导出再拼第三份”，而是尽量围绕同一份稳定正文源工作，减少所见非所得。

### 3. 公式进入正式文档链路

公式型内容不是最后补救的附件。系统会尽量让公式从正文阶段就进入正式文档链，而不是交稿前再人工重做。

### 4. Word 正式交付导向

产品重点不是写完即止，而是继续面向 Word 等正式交付格式，尽量保留结构、段落、样式和可继续编辑性。

### 5. 本地优先与高敏感场景适配

资料校验、解析和部分运行链路优先在本地完成，更适合对资料敏感度较高的工作场景。

### 6. 失败优先中止而不是伪成功

当文件链路、解析或关键生成环节出现异常时，系统优先中止并报错，而不是继续伪造“已经成功读取/生成”的结果。

## 它更适合的任务

- 正式报告
- 研究说明
- 课程材料
- 公式型内容文档
- 需要 Word 继续编辑的交付稿

## 它不主打的任务

- 只要几句灵感的随手问答
- 纯聊天式头脑风暴
- 当前就要求完整公开跨平台客户端的场景

## 能力边界

DeepDcc 当前公开测试版强调的是“文档生产工作流”，不是“万能办公自动化代理”。

它当前更关注：

- 从任务到蓝图再到正文的收口
- 正文、预览、导出的交付一致性
- 公式型与长文档型内容的正式链路

而不是优先承诺：

- 全平台客户端覆盖
- 完整团队协作后台
- 已经公开上线的正式收费体系
