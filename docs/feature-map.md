# 特色功能总览

这页不是功能清单，而是 DeepDcc 当前最值得理解的几项“工作流级能力”。

## 1. 蓝图先行，而不是直接放正文

很多写作工具的默认路径是：

- 输入一句话
- 直接吐出一大段正文
- 然后让你自己收拾结构

DeepDcc 更强调先生成蓝图、先确认章节，再释放正文。这一步的意义不是“多一个步骤”，而是让长文档在一开始就有约束边界。

**价值在于：**
- 长文不容易越写越散
- 章节可改、可重生成、可确认
- 用户更容易控制最终交付结构

## 2. 正文、预览、导出尽量围绕同一份稳定内容源

DeepDcc 不追求“编辑器一份、预览一份、导出再拼一份”的三套内容状态，而是尽量让它们围绕同一份稳定正文源工作。

**价值在于：**
- 降低所见非所得
- 预览不是脱离正文的独立假象
- 导出更接近用户已经确认过的内容

## 3. 公式不是外挂，而是正式文档链的一部分

对于公式型内容，很多工具只能给出 LaTeX 源码、截图或临时渲染结果，真正进入文档交付时仍然需要人工补救。

DeepDcc 的方向，是让公式型内容尽量从正文阶段就进入正式文档链，而不是最后临门一脚再修补。

**价值在于：**
- 公式型主题不必完全靠手工收尾
- 预览与最终交付更容易对齐
- 更适合研究说明、课程材料、技术文档

## 4. 目标不是“生成一段像样文字”，而是“能交稿”

DeepDcc 更在意的是：

- 结构是否可控
- 预览是否可信
- 导出是否还能继续编辑
- 最终文档是否适合进入正式交付流程

所以它更像“文档生产工作台”，而不是只比较模型回答质量的聊天写作器。

## 5. 本地优先与失败可见

当资料解析、校验或关键生成链路出现异常时，DeepDcc 更强调失败可见，而不是继续伪造一个看起来已经成功的结果。

**价值在于：**
- 更适合高敏感度资料场景
- 更容易暴露真实问题
- 用户不会被假成功误导到后续交付阶段

## 一句话总结

DeepDcc 当前最值得理解的，不是某一个按钮，而是它在尝试把“任务理解 -> 蓝图 -> 正文 -> 预览 -> 导出”变成一条更适合正式文档交付的连续链路。
