# 用户路径说明

这页不是系统内部流程，而是从用户视角描述：你会做什么，系统会给你什么，你下一步该看什么。

## 路径 1：从一个想法到一份正式报告

### 第一步：输入任务

你输入需求，必要时再挂载资料。

**你看到的是：**
- 一个明确的写作起点
- 资料可以参与后续生成的入口

**你下一步看什么：**
- 系统是否正确理解了任务类型和目标

## 第二步：确认智能解析结果

系统先判断这是报告、说明、方案还是其他结构化文档，而不是立刻整篇输出。

**你看到的是：**
- 一次任务理解确认
- 风格、主题和结构倾向的初步判断

**你下一步看什么：**
- 它理解得对不对
- 是否适合继续生成蓝图

## 第三步：查看蓝图

系统给出章节结构。这里的重点不是“它替你想好了”，而是“它先把结构摊开给你看”。

**你看到的是：**
- 章节列表
- 可修改的标题
- 可重新生成的大纲

**你下一步看什么：**
- 章节顺序是否合理
- 有没有缺失或冗余章节

## 第四步：生成正文

确认蓝图后，系统再开始展开正文。

**你看到的是：**
- 一份按蓝图展开的长文稿
- 更接近交付文档的段落结构

**你下一步看什么：**
- 内容是否跑偏
- 结构是否和蓝图一致

## 第五步：质量审核

在真正把这份文稿当作“完成”之前，系统还会经过质量审核。

**你看到的是：**
- 完成前的收尾与修整
- 更稳定的正文结果

**你下一步看什么：**
- 最终结果是否已经达到可预览、可核对状态

## 第六步：打开预览

预览阶段不是装饰，而是交付前确认结构、分页和视觉组织的重要环节。

**你看到的是：**
- 更接近最终导出的视图
- 用于核对结构和分页的版面表现

**你下一步看什么：**
- 这是不是你愿意拿去导出的结果

## 第七步：导出并继续交付

当正文与预览都确认后，再进入正式导出链路。

**你得到的是：**
- 更适合继续交付、继续编辑或继续协作的正式文档结果

## 这条路径的关键点

DeepDcc 的价值不在某一个单点功能，而在于每一步之间都有承接关系：

- 不是解析完就忘了结构
- 不是蓝图确认后就不管正文
- 不是正文写完就不管预览
- 不是预览看过就不管导出

它试图让用户始终知道：

**现在处在哪一步，下一步该看什么，最终结果会不会偏。**
